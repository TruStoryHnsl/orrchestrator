# /bugfix-analyze — Architectural analysis of recorded bugfixes

You are a shrewd systems architect analyzing recorded bugs and their solutions. Your job is to extract architectural insights that help the dev team understand *why* these bugs exist at a deeper level — not just what broke, but what structural patterns, edge-case interactions, or design decisions made them possible.

## Input

1. If `$ARGUMENTS` names a project, read `~/projects/<project>/.bugfix-ledger.md`.
   If no arguments, read `~/projects/.bugfix-ledger.md` (umbrella/admin/infra).

2. Also read relevant source code, configs, and architecture docs:
   - The project's `CLAUDE.md` if it exists
   - Source files referenced in bug entries
   - Related config files (systemd units, Docker compose, etc.)
   - The umbrella `~/projects/CLAUDE.md` for cross-project context

## Analysis process

For each bug (or cluster of related bugs), perform this analysis:

### 1. Architectural classification
Classify each bug into one or more categories:
- **State lifecycle mismatch** — Component A assumes state that component B has invalidated
- **Permission/capability boundary** — Access control doesn't survive a state transition
- **Resource lifecycle gap** — Resource acquired in one phase, needed in another, but not preserved
- **Recovery path absence** — Happy path works, but no graceful recovery from interruption
- **Configuration drift** — Runtime state diverges from declared configuration
- **Coupling through shared mutable state** — Components interact through implicit shared state
- **Abstraction leak** — Lower-level details (hardware, OS, driver) break through abstractions
- **Concurrency/ordering** — Timing-dependent failures between async components

### 2. Interaction mapping
Map which system layers are involved:
```
[Hardware/Driver] → [Kernel/OS] → [Systemd/Init] → [Desktop/App] → [User]
```
Identify exactly where the failure occurs in this chain and what assumptions broke.

### 3. Pattern recognition
Look across all recorded bugs for:
- Recurring root causes (same underlying issue manifesting differently)
- Common trigger patterns (sleep, restart, network change, etc.)
- Clusters of related failures that suggest a deeper structural issue
- Solutions that are band-aids vs. solutions that address root causes

### 4. Preventive architecture recommendations
Based on patterns found, suggest:
- Monitoring/alerting that would catch issues earlier
- Architectural changes that would eliminate categories of bugs
- Defensive patterns that should be standard practice

## Output format

Write the companion report to the same directory as the ledger, named `.bugfix-analysis.md`.

```markdown
# Bugfix Architectural Analysis — <project or "Infrastructure">

> Auto-generated by /bugfix-analyze. Last updated: <YYYY-MM-DD>
> Source: .bugfix-ledger.md (<N> entries analyzed)

## Executive summary
<2-3 sentences: what patterns emerge, what the biggest risks are>

## Bug taxonomy

### <Category name>
**Bugs:** BUG-xxxx-xx-xxx, BUG-xxxx-xx-xxx
**Pattern:** <describe the shared architectural pattern>
**Interaction layers:** <which layers are involved>
**Why this category exists:** <structural reason this class of bug is possible>
**Recommended mitigation:** <architectural change or defensive pattern>

<repeat for each category>

## Cross-cutting patterns
<patterns that span multiple categories>

## Architectural risk map
<which system boundaries or transitions are most fragile>

## Recommendations
<prioritized list of architectural improvements, ordered by impact>
```
