A — Terminal Stream (xterm.js)

Pipe the actual TUI through a PTY → WebSocket → xterm.js in the browser. The web UI IS the TUI — pixel-perfect, instant feature parity, zero drift. Mobile adds a virtual keyboard overlay that sends key events.

Pros

  • 100% feature parity immediately
  • No separate UI to maintain
  • Every TUI update is free

Cons

  • Browser renders terminal — not "web-native"
  • Mobile control pad sends key codes, not touch
  • No web-specific improvements (hyperlinks, rich media)

B — Native Web UI (React + WebSocket)

Rebuild the TUI as a proper web app. Rust backend exposes a WebSocket API that streams app state as JSON. React frontend renders it natively — clickable items, hover states, rich text, drag-and-drop.

Pros

  • True web interactions (click, hover, drag)
  • Mobile-native touch controls
  • Can add features TUI can't do

Cons

  • Significant build — recreate all TUI panels
  • State sync complexity
  • Two UIs to maintain

C — Hybrid (xterm.js shell + web overlays)

Start with xterm.js streaming for immediate parity. Add a thin web layer on top: clickable hit regions mapped to key presses, hover tooltips, and progressive web enhancements panel by panel over time.

Pros

  • Ships fast with full parity
  • Web improvements are additive
  • Best of both

Cons

  • Two rendering systems to sync
  • Overlay hit-region mapping is fiddly

My recommendation: A (terminal stream) — feature parity is the stated goal, and the mobile control pad maps perfectly to keyboard events. Native web improvements can be layered in later if wanted.
