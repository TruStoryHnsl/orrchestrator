reload "where am I" before pressing anything. Every entry here is verified

**Subtler conflicts:**

- `Esc` has at least four distinct behaviors depending on subview:
  closes intake review (`crates/orrch-tui/src/app.rs:2009`), opens app
  menu from List (`crates/orrch-tui/src/app.rs:2047`), closes app menu
  (`crates/orrch-tui/src/app.rs:2055`), or is forwarded to a tmux session
  in `ExpandedSession` (`crates/orrch-tui/src/app.rs:2040`). The
  ExpandedSession variant uses a "double-Esc within 500ms" gesture
  (`crates/orrch-tui/src/app.rs:932-934`) — the only place double-key
  timing is meaningful in the TUI.
- Tab in Oversee literally does nothing
  (`crates/orrch-tui/src/app.rs:3795-3797` — comment: "Tab now does
  nothing — use → to enter projects"). It's dead-key territory; no hint
  to the user that → is the actual entry mechanism.
- Mouse-click on the WebUI badge area is dispatched as a separate path
  (`crates/orrch-tui/src/app.rs:919-923`) but the badge is no longer
  rendered (per CLAUDE.md "no longer surfaced as a status-bar badge");
  the click handler is presumably orphaned. Unverified — needs a grep
  for `webui_badge_area`.

## 3. Discoverability Gaps

Features that exist, work, and are essentially invisible without reading
the source.

1. **`V` (visibility) in Oversee** — newly added by VIS-001 in this
   sprint. Hint added to the Oversee hint line
   (`crates/orrch-tui/src/ui.rs:3814`) and a `(N hidden, V to toggle)`
   badge appears in the Projects block title only when something is
   actually hidden (`crates/orrch-tui/src/ui.rs:projects_title`). Until
   the user hides a scope, they will only see it via the hint line.
2. **`Ctrl+w` in Design > Workforce** launches the web node editor in
   the system browser (`crates/orrch-tui/src/app.rs:2020-2028`). The
   `Ctrl+w` shortcut isn't in any hint line; the only signal is the
   CLAUDE.md mention.
3. **`Ctrl+u` opens the WebUI in browser + copies URL** globally from
   any panel (`crates/orrch-tui/src/app.rs:2031-2034`). No hint line
   mentions it.
4. **Double-Esc in Hypervise > ExpandedSession** exits expanded mode;
   single Esc forwards to the session
   (`crates/orrch-tui/src/app.rs:2040-2042` with state at lines 932-934).
   No on-screen hint mentions the timing.
5. **Color-tag cycle on `t`** in Oversee
   (`crates/orrch-tui/src/app.rs:3785-3793`) — cycles
   None→Red→Yellow→Green→None. Hint line has no `t` entry.
6. **`P` in Oversee opens the new-project wizard**
   (`crates/orrch-tui/src/app.rs:3736`) — completely missing from the
   Oversee hint line.
7. **`f` in Oversee opens a global feedback editor**
   (`crates/orrch-tui/src/app.rs:3782`). Not in hints.
8. **Renaming via `r`** when a project is selected vs. **reloading via
   `r`** when none is selected (`crates/orrch-tui/src/app.rs:3767-3781`)
   — same key, two different actions, no on-screen indicator of which
   mode you're in.
9. **`g` runs `git commit + push` for the selected project**
   (`crates/orrch-tui/src/app.rs:3747-3766`). Not in hint lines.
10. **`Shift+Left/Right` switches panels and lands at content depth**
    (`crates/orrch-tui/src/app.rs:1999-2005`). The status bar's
    direction-key hints don't disclose Shift modifier semantics.

**Inventory observations:**
- 24 distinct popup states. ~6 of them share the rename popup renderer;
  the rest are bespoke. The `draw_*` fns duplicate a lot of the
  `centered_popup → Clear → render Paragraph in styled Block` pattern —
  prime candidate for a `Modal` helper in OPT-020b.
- Confirm dialogs use varying y/n vs. y/Enter semantics. Examples:
  `key_confirm_complete` accepts y/Enter (`crates/orrch-tui/src/app.rs`
  inside the match at line 2169-2172 dispatch), while
  `key_confirm_delete_deprecated` only accepts y/Y
  (`crates/orrch-tui/src/app.rs:2334`). Inconsistent.
- Several popups (e.g. `RoutingSummary`) accept any key to dismiss —
  surprising if users expect Esc-only. The intentions of "any key
  acknowledges" vs. "explicit Esc" aren't documented anywhere visible.

## 5. State Drift Hazards

Places where two pieces of state can disagree because the relationship is
imperative and not enforced by a type.

1. **`project_selected` (a flat-list cursor) vs. `build_list_map()`
   (the actual flat list).** Whenever the visible set changes —
   reload, deprecate, complete, scope-visibility toggle, deprecated
   folder appearing/disappearing — the cursor can point past the end
   or at a section header. `build_list_map` and `selected_project_index`
   (`crates/orrch-tui/src/app.rs:1267-1272`) silently return `None` on
   mismatch. VIS-001 added a clamp inside `toggle_scope_visibility`
   (this sprint) but the same hazard existed before for `reload_projects`
   / `deprecate` / `complete` paths. Search-and-fix required for
   OPT-020b. References: `crates/orrch-tui/src/app.rs:616,1227-1262`.
2. **`hot_indices`/`cold_indices`/`ignored_indices` cached lists vs.
   underlying `projects` Vec.** `categorize_projects` rebuilds them
   (`crates/orrch-tui/src/app.rs:1306-1320`) but is called explicitly —
   not derived. Any code path that mutates `projects[i]` (e.g. setting
   `temperature`, deprecating, the new VIS-001 scope hide) MUST remember
   to call `categorize_projects()` or the cached vectors lie. Currently
   relies on `key_projects()` calling it on every key press
   (`crates/orrch-tui/src/app.rs:3687`).
3. **`hidden_scopes` on `App` vs. on disk (`Config`).** VIS-001's
   `save_hidden_scopes()` reloads Config-from-disk and overwrites only
   the `hidden_scopes` field
   (`crates/orrch-tui/src/app.rs:save_hidden_scopes`). If two
   orrchestrator processes run concurrently (which the CLAUDE.md
   explicitly warns against), the last writer wins — silently. Same
   hazard exists for `Valve::save` and the `show_deprecated` flag
   (which doesn't persist at all and so can't drift, but is
   inconsistent with VIS-001's persistence model).
4. **`tree_browsing` / `tree_project` / `tree_selected` triple.**
   These can disagree (e.g. `tree_browsing = true` but `tree_project = None`)
   — would yield the no-op early-return in `key_projects`
   (`crates/orrch-tui/src/app.rs:3691-3695`). Currently only set
   together but not enforced.
5. **`focus_depth` vs. panel content depth.** `content_depth()` is
   implicit — every key handler that wants to know "are we at content
   level" calculates it differently. Examples in `Shift+Left/Right`
   (`crates/orrch-tui/src/app.rs:2002-2003`) and the focus_depth set
   on startup (`crates/orrch-tui/src/app.rs:992`).
6. **`spawn_*` wizard fields (`spawn_project_idx`, `spawn_goal_text`,
   `spawn_agent_idx`, etc.) at `crates/orrch-tui/src/app.rs:643-651`.**
   Persistent across the lifetime of the App — opening the spawn wizard
   in different contexts can land on the previously-typed goal. Whether
   that's a feature or a bug depends on user expectation (it surprised
   me in 30s of reading).
7. **`expanded_pane_content` vs. live tmux pane.** Capture is
   throttled to 10 Hz (`crates/orrch-tui/src/app.rs:929-930`); on rapid
   user typing, the user sees their own keystrokes echo with up to a
   100ms lag. Acceptable but worth noting.
8. **`webui_token` is also stored in `Config` (presumably) AND in env
   vars.** The TUI reads from `App.webui_token`
   (`crates/orrch-tui/src/app.rs:917`); environment changes after
   process start are invisible. Likely fine for normal use but breaks
   `setcap`/restart-style workflows.

## 6. Recommended Refactor Priorities for OPT-020b

Ranked by user-visible impact / effort ratio. Effort: S (1-2 hrs), M
(half-day), L (1+ day).

1. **[S] Add a "show all keybindings" overlay** keyed off `?`. The
   current hint-line approach can only fit ~5 keys; the actual
   per-panel set is 8-15 keys. A modal that lists all bindings for
   the current panel, autogenerated from a single registry, would
   fix Discoverability gap items 1-10 in one stroke. Bind from any
   panel.
2. **[S] Convert ad-hoc popup `draw_*` fns to a shared `draw_modal(title,
   body_lines, hints)` helper.** `crates/orrch-tui/src/ui.rs` has ~24
   modals; ~80% follow an identical pattern (centered_popup, Clear,
   styled bordered Block, body Paragraph, hint footer). Extract this
   once; every new modal gets consistent visuals + free Esc handling.
3. **[S] Standardize confirm dialogs on `y/Enter = confirm, n/Esc =
   cancel` everywhere.** Currently `key_confirm_delete_deprecated`
   only accepts y/Y (`crates/orrch-tui/src/app.rs:2334`); other
   handlers accept y/Enter (`crates/orrch-tui/src/app.rs:2173-2188`
   dispatch chain). Pick one and apply consistently.
4. **[M] Replace `project_selected` (flat cursor) with a `Selection`
   enum** (`Project(idx)`, `Production(idx)`, `Facility(fi, si)`,
   `DeprecatedFolder`, `Section(label)`). Eliminates the
   "selected_project_index returns None" surprise across all panels
   and removes the manual clamping after every list mutation.
5. **[M] Make `hot_indices`/`cold_indices`/`ignored_indices` derived
   on demand** (cheap iterator over `projects`) instead of cached.
   Removes the "forgot to call categorize_projects" footgun (state
   drift hazard #2). Profile-check first — `projects.len()` is
   typically <50, iteration cost is sub-microsecond.
6. **[M] Unify the keybinding letter mapping into a per-panel table**
   resolved at `handle_key`-time. Today, every panel has its own giant
   match arm with literal character checks; conflicts (Section 2
   above) are invisible without grep. A declarative table makes
   conflicts surface at compile time (assertion on duplicates) AND
   feeds the help overlay (item 1) for free.
7. **[M] Persist `show_deprecated` in `Config` for symmetry with
   VIS-001's `hidden_scopes`** (`crates/orrch-tui/src/app.rs:621`).
   Currently reset every launch.
8. **[L] Replace the `SubView` variant explosion (24 popups) with a
   modal stack** (`Vec<Box<dyn ModalState>>`). Lets popups compose
   (e.g. confirm-on-rename), simplifies the giant `match` in
   `handle_key`, and makes the modal-vs-non-modal distinction
   explicit instead of inferred from variant name.
9. **[L] Split `app.rs` into per-panel modules** (`oversee.rs`,
   `design.rs`, `hypervise.rs`, `analyze.rs`, `publish.rs`). At 7554
   LOC, `app.rs` is the largest file in the workspace; cross-panel
   conflicts (Section 2) are only visible when you read the whole
   thing. Same for `ui.rs` at 4466 LOC.
10. **[L] Centralize tmux pane capture into a small actor** rather
    than the throttled-poll-from-render-thread pattern at
    `crates/orrch-tui/src/app.rs:929-934`. Render becomes pure;
    capture lag becomes tunable per-window.

## Appendix — Methodology

This audit is based on direct reading of:

- `crates/orrch-tui/src/app.rs` (7554 LOC)
- `crates/orrch-tui/src/ui.rs` (4466 LOC)
- `crates/orrch-tui/src/dashboard.rs` (2 LOC, placeholder)

audit using a real terminal session and screenshot capture would be
worth the half-day to confirm/refute the discoverability claims in
Section 3 from a real-eyes perspective (compliant with the CLAUDE.md
"observe the result of your work" rule).
